# سیاست عملیاتی چرخه عمر دسترسی — نسخه ۰ (پیش‌نویس)

**سامانه:** MOGHARE360 ERP  
**حوزه:** مدیریت دسترسی و چرخه عمر کاربر  
**وضعیت سند:** پیش‌نویس عملیاتی نسخه ۰  
**اصل بنیادی:** هیچ کاربری مجوز مستقیم و تصادفی دریافت نمی‌کند. دسترسی فقط از مسیر **نقش** و از طریق **درخواست کنترل‌شده، تأیید، و اعمال رسمی** داده یا گرفته می‌شود.

---

## ۱. چه کسی می‌تواند درخواست دسترسی ثبت کند؟

درخواست دسترسی توسط افراد زیر مجاز است، مشروط به داشتن مجوز `access.request.create` در سامانه:

| نقش / واحد | دامنه درخواست |
|------------|----------------|
| **منابع انسانی (HR)** | استخدام جدید، خروج پرسنل، تغییر واحد سازمانی |
| **مدیر واحد (Department Manager)** | نقش، ارتقا، تنزل، تعلیق، محدودیت، دسترسی موقت برای پرسنل همان واحد |
| **مدیر عملیات (Operations Manager)** | درخواست‌های بین‌واحدی و ارتقای سطح بالا |
| **مدیر سیستم (System Admin)** | همه انواع درخواست برای همه پرسنل (به‌جز نقش مالک) |
| **مالک سیستم (Owner)** | همه انواع درخواست؛ در حالت اضطراری بدون نیاز به تأیید دیگران (با ثبت اجباری دلیل) |
| **پرسنل عادی (Staff)** | فقط درخواست برای **خود** و فقط از نوع دسترسی موقت یا نقش پیشنهادی محدود؛ نیازمند تأیید مدیر واحد |

**قواعد ثبت درخواست:**
- هر درخواست باید شامل **نوع درخواست**، **کاربر هدف**، **نقش(های) پیشنهادی** (نه مجوز مستقیم)، **توضیح و دلیل**، و در صورت نیاز **تاریخ شروع** و **تاریخ پایان** باشد.
- درخواست‌های **فوری (urgent)** فقط با **اطلاع مالک سیستم** قابل ثبت است.
- ثبت درخواست به‌معنای اعطای دسترسی نیست؛ تا زمان تأیید و اعمال، دسترسی تغییر نمی‌کند.

---

## ۲. چه کسی می‌تواند دسترسی را تأیید کند؟

تأیید دسترسی فقط توسط افراد دارای مجوز `access.request.approve` و مطابق **ماتریس تأیید نسخه ۰** انجام می‌شود:

| نوع درخواست | حداقل تأییدکنندگان |
|-------------|-------------------|
| **استخدام / ورود (ONBOARDING)** | مدیر واحد + مدیر سیستم |
| **اعطای نقش (ROLE_GRANT)** | مدیر واحد |
| **دسترسی موقت (TEMPORARY_ROLE_GRANT)** | مدیر واحد |
| **ارتقای دسترسی (ACCESS_UPGRADE)** | مدیر واحد + مدیر سیستم |
| **تنزل دسترسی (ACCESS_DOWNGRADE)** | مدیر واحد |
| **ارتقای شغلی (PROMOTION)** | مدیر واحد + مدیر عملیات |
| **تعلیق (SUSPENSION)** | مدیر واحد + مدیر سیستم |
| **محدودیت خطا/تخلف (ACCESS_RESTRICTION)** | مدیر واحد + مدیر سیستم |
| **خروج پرسنل (OFFBOARDING)** | HR یا مدیر واحد + مدیر سیستم |
| **اضطراری (Owner)** | خود مالک (با ثبت اجباری دلیل در لاگ) |

**محدوده تأیید مدیر واحد:** فقط برای پرسنلی که `department_id` آن‌ها با واحد مدیر یکسان است.

**تصمیم‌های مجاز:** تأیید کامل، تأیید جزئی، رد، یا بازگرداندن برای اصلاح. هر تصمیم باید **یادداشت تأییدکننده** داشته باشد.

**در صورت عدم دسترسی تأییدکننده:** پس از مدت تعیین‌شده (در نسخه ۰ به‌صورت دستی)، مدیر سیستم می‌تواند درخواست را ارجاع یا رسیدگی کند.

---

## ۳. چه کسی می‌تواند دسترسی را فعال (اعمال) کند؟

**فعال‌سازی** مرحله‌ای جدا از تأیید است و فقط پس از رسیدن درخواست به وضعیت `APPROVED` یا `PARTIALLY_APPROVED` انجام می‌شود.

| نقش | مسئولیت فعال‌سازی |
|-----|------------------|
| **سیستم (خودکار)** | پس از تکمیل زنجیره تأیید، اعمال خودکار نقش‌ها، به‌روزرسانی وضعیت چرخه عمر، و ثبت تاریخچه |
| **مدیر سیستم** | اعمال دستی در صورت نیاز، رسیدگی به درخواست‌های نیمه‌تأییدشده، و اعمال پس از مهاجرت |
| **مالک سیستم** | اعمال فوری در حالت اضطراری (همراه با ثبت audit) |

**پس از فعال‌سازی:**
- نقش(ها) در جدول نقش‌های کاربر با `effective_from` ثبت می‌شوند.
- وضعیت چرخه عمر کاربر به‌روز می‌شود.
- تاریخچه تغییر دسترسی و لاگ audit ثبت می‌شود.
- درخواست به وضعیت `APPLIED` منتقل می‌شود.

**هیچ فردی — از جمله مدیر سیستم — حق ندارد نقش را مستقیماً در پایگاه داده یا از طریق فرم کاربران بدون درخواست تأییدشده و اعمال رسمی اضافه کند** (به‌جز مهاجرت اولیه مستند و override اضطراری مالک).

---

## ۴. مسیر دسترسی استخدام پرسنل جدید (Onboarding)

```
پیش‌نویس کاربر → درخواست ONBOARDING → تأیید → اعمال → ACTIVE
```

**گام‌ها:**

1. **HR یا مدیر سیستم** کاربر را در وضعیت `DRAFT` ایجاد می‌کند (نام، واحد، سمت، تاریخ شروع).
2. در این مرحله **ورود به سامانه ممنوع** است.
3. **درخواست ONBOARDING** ثبت می‌شود؛ نقش(های) پیشنهادی بر اساس سمت یا انتخاب سیاست‌محور (نه خودکار).
4. **مدیر واحد** و سپس **مدیر سیستم** درخواست را بررسی و تأیید می‌کنند.
5. پس از اعمال:
   - وضعیت کاربر → `ACTIVE`
   - ورود فعال می‌شود
   - نقش‌ها از تاریخ `effective_from` (معمولاً تاریخ شروع همکاری) اثر می‌گیرند
   - تاریخچه onboarding در سامانه ثبت می‌شود

**نکته:** تغییر واحد یا سمت به‌تنهایی نقش را عوض نمی‌کند؛ در صورت نیاز درخواست جداگانه لازم است.

---

## ۵. مسیر دسترسی ارتقای شغلی (Promotion)

```
درخواست PROMOTION → تأیید مدیر واحد + مدیر عملیات → اعمال
```

**محتوای درخواست:**
- سمت جدید (الزامی)
- واحد جدید (در صورت انتقال)
- نقش(های) جدید (اعطا)
- نقش(های) قبلی (لغو — اختیاری ولی توصیه‌شده)

**تأیید:** مدیر واحد + مدیر عملیات (سطح بالاتر از اعطای نقش عادی).

**پس از اعمال:** وضعیت کاربر می‌تواند `PROMOTED` یا `ACCESS_CHANGED` ثبت شود؛ تاریخچه قبل/بعد ذخیره می‌شود.

---

## ۶. مسیر تنزل دسترسی (Access Downgrade)

```
درخواست ACCESS_DOWNGRADE → تأیید مدیر واحد → اعمال (فوری یا زمان‌بندی‌شده)
```

**شروع‌کنندگان مجاز:** مدیر واحد، HR، مدیر سیستم.

**کاربرد:** حذف نقش اضافی، کاهش سطح پس از تغییر وظیفه، یا اصلاح دسترسی بیش از حد.

**مسیر سریع‌تر:** نسبت به ارتقا، تنها یک تأیید (مدیر واحد) کافی است.

**اثر:** نقش(های) مشخص‌شده `revoked_at` می‌گیرند یا در تاریخ آینده منقضی می‌شوند. جلسه فعال کاربر در صورت لزوم باطل می‌شود.

---

## ۷. مسیر دسترسی موقت (Temporary Access)

```
درخواست TEMPORARY_ROLE_GRANT → تأیید مدیر واحد → اعمال با expires_at اجباری
```

**قواعد:**
- **تاریخ پایان (`expires_at`) اجباری** است؛ بدون آن درخواست پذیرفته نمی‌شود.
- در رابط کاربری با برچسب **«موقت»** نمایش داده می‌شود.
- پس از انقضا، نقش به‌صورت خودکار یا از طریق بررسی روزانه (در نسخه ۰: روش مستند دستی/زمان‌بند) لغو می‌شود.
- تمدید فقط با **درخواست جدید** ممکن است، نه ویرایش خام تاریخ.

**مثال:** دسترسی موقت انبار برای پذیرش‌گر در شیفت خاص، یا دسترسی گزارش برای مدیر در دوره ممیزی.

---

## ۸. مسیر تعلیق دسترسی (Suspension)

```
درخواست SUSPENSION → تأیید مدیر واحد + مدیر سیستم → اعمال تعلیق
```

**دامنه تعلیق (یکی از موارد):**
- `FULL` — مسدودسازی کامل دسترسی
- `MODULE` — مسدودسازی یک یا چند ماژول
- `LOGIN_ONLY` — فقط ورود ممنوع

**ثبت اجباری:** دلیل، تاریخ شروع، تاریخ پایان (اختیاری برای تعلیق بازه‌ای).

**اثر:** حتی اگر نقش فعال باشد، در بازه تعلیق `accessHas` برای دامنه مربوط **رد** می‌شود. وضعیت چرخه عمر → `SUSPENDED`.

**رفع تعلیق:** فقط از طریق درخواست جدید (مثلاً لغو تعلیق یا پایان خودکار `ends_at`).

**اضطرار:** مالک می‌تواند فوراً تعلیق کند؛ باید درخواست بازگشتی (retroactive) و دلیل در audit ثبت شود.

---

## ۹. مسیر محدودیت ناشی از خطا یا تخلف (Access Restriction)

```
درخواست ACCESS_RESTRICTION → تأیید مدیر واحد + مدیر سیستم → اعمال محدودیت هدفمند
```

**تفاوت با تعلیق:** محدودیت **باریک‌تر** است — فقط ماژول یا مجوز(های) مشخص مسدود می‌شوند، نه لزوماً کل حساب.

**ثبت اجباری:** شرح حادثه / خطا / تخلف (incident note).

**کاربرد:** اشتباه در ثبت مالی، نقض سیاست امنیتی، دسترسی اشتباه به قیمت خرید، و موارد مشابه.

**وضعیت چرخه عمر:** `RESTRICTED` (در کنار نقش‌های باقی‌مانده).

**رفع محدودیت:** درخواست جداگانه پس از بررسی و تأیید.

---

## ۱۰. مسیر خروج پرسنل (Offboarding)

```
درخواست OFFBOARDING → تأیید HR/مدیر واحد + مدیر سیستم → OFFBOARDING → INACTIVE
```

**گام‌ها:**

1. ثبت درخواست خروج با دلیل (استعفا، پایان قرارداد، اخراج، انتقال).
2. تأیید HR یا مدیر واحد + مدیر سیستم.
3. وضعیت → `OFFBOARDING`: **هیچ اعطای جدیدی** مجاز نیست؛ لغو نقش‌ها در صف قرار می‌گیرد.
4. پس از تکمیل: وضعیت → `INACTIVE`، همه نقش‌ها لغو، ورود غیرفعال، جلسات باطل.
5. دوره خنک‌سازی (اختیاری): حساب وجود دارد اما ورود ممنوع است.

**اصل:** خروج پرسنل بدون درخواست و audit رسمی **ممنوع** است.

---

## ۱۱. بازنویسی اضطراری مالک (Emergency Owner Override)

**تنها استثنای کنترل‌شده** برای عبور از زنجیره تأیید.

**مجاز برای:** مالک سیستم (`is_system_owner`).

**موارد استفاده:**
- تعلیق فوری کاربر پرخطر
- بازگردانی دسترسی حیاتی در outage عملیاتی
- قفل اضطراری پس از نقض امنیتی

**الزامات:**
- ثبت **دلیل اجباری** در audit
- ثبت در `core_audit_logs` با برچسب emergency
- در صورت امکان، ثبت **درخواست بازگشتی** (retroactive) در همان روز کاری
- اقدامات حساس مالک نیز audit می‌شوند

**ممنوع:** استفاده روزمره برای دور زدن فرآیند تأیید مدیر واحد.

---

## ۱۲. سوابق audit اجباری

هر رویداد زیر **باید** ثبت شود:

### در `core_audit_logs` (لاگ امنیتی عمومی)
- ثبت، ارسال، لغو، تأیید، رد درخواست دسترسی
- تغییر وضعیت چرخه عمر کاربر
- اعمال نقش (grant/revoke)
- ایجاد/رفع تعلیق و محدودیت
- انقضای دسترسی موقت
- ورود ناموفق و رد دسترسی به صفحه (در صورت پیکربندی)
- override اضطراری مالک
- استفاده از fallback دسترسی legacy (موقت تا پایان مهاجرت)

### در `core_access_change_history` (تاریخچه تغییر دسترسی)
- هر نتیجه تأییدشده با **snapshot قبل و بعد** (JSON)
- شناسه درخواست مبدأ (`request_id`)
- نوع تغییر، زمان، عامل اعمال

### در `core_access_approvals`
- هر تصمیم تأییدکننده: تأیید / رد / جزئی + یادداشت + زمان

**اصل:** تاریخچه تغییر دسترسی **غیرقابل حذف** است. اصلاح فقط با درخواست جدید و رکورد جدید انجام می‌شود.

## ۱۳. موارد ممنوع

| ممنوعیت | توضیح |
|---------|--------|
| **اعطای مستقیم مجوز به کاربر** | مجوز فقط به نقش متصل می‌شود؛ جدول user-permission وجود ندارد |
| **تغییر دسترسی بدون درخواست** | هیچ UI یا SQL مستقیمی برای `core_user_roles` بدون درخواست `APPLIED` |
| **ورود پرسنل غیر ACTIVE** | به‌جز سیاست مالک |
| **نقش بدون تاریخ شروع** | هر اعطا باید `effective_from` داشته باشد |
| **دسترسی موقت بدون تاریخ پایان** | `expires_at` اجباری |
| **تعلیق/محدودیت بدون دلیل** | فیلد justification/incident اجباری |
| **خروج پرسنل بدون offboarding** | لغو دسترسی خام بدون workflow ممنوع |
| **تأیید خارج از دامنه واحد** | مدیر واحد فقط برای پرسنل همان department |
| **درخواست فوری بدون اطلاع مالک** | urgent بدون acknowledgment مالک |
| **ویرایش یا حذف audit و تاریخچه** | immutable |
| **استفاده از پروفایل دسترسی legacy برای اعطای جدید** | فقط خواندن موقت در bridge؛ اعطای جدید فقط از workflow |
| **دستکاری مستقیم پایگاه داده تولید** | بدون backup، مستندسازی، و تأیید دستی |
| **دور زدن روزمره override مالک** | فقط اضطرار واقعی |

## پیوست: وضعیت‌های چرخه عمر کاربر

| وضعیت | معنی | دسترسی |
|-------|------|--------|
| `DRAFT` | پیش‌نویس | بدون ورود |
| `PENDING_ONBOARDING` | در انتظار تأیید استخدام | بدون ورود |
| `ACTIVE` | فعال | فقط نقش‌های تأییدشده |
| `SUSPENDED` | تعلیق | رد بر اساس دامنه تعلیق |
| `RESTRICTED` | محدودیت خطا/تخلف | رد در ماژول/مجوزهای محدودشده |
| `OFFBOARDING` | در حال خروج | بدون اعطای جدید |
| `INACTIVE` | پایان همکاری | بدون نقش، بدون ورود |

---

## پیوست: جریان کلی درخواست

```
DRAFT → SUBMITTED → UNDER_REVIEW → APPROVED / PARTIALLY_APPROVED / REJECTED → APPLIED
```

- **REJECTED:** بدون تغییر دسترسی؛ دلیل رد در audit.
- **CANCELLED:** فقط توسط ثبت‌کننده قبل از اعمال.
- **PARTIALLY_APPROVED:** فقط بخش تأییدشده اعمال می‌شود.
